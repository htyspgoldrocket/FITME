# CLAUDE.md — FITME

> AI 가상 피팅 모바일 앱. Claude Code가 이 파일을 **매 세션 시작 시 먼저 읽고** 개발을 진행한다.
> 이 문서는 프로젝트의 **단일 진실 공급원(Single Source of Truth)** 이다.

## 📌 0. 절대 규칙 (Working Rules) — 가장 먼저, 항상 지킬 것

Claude Code는 아래 규칙을 **모든 작업에서 무조건 우선** 적용한다.

### 🚨 규칙 1 — 정직성과 즉시 보고 (가장 중요, 최우선)

> **작동하지 않는 코드를 절대 작동하는 것처럼 위장하지 않는다.**

- 코드를 작성하면 **항상 실제로 실행해서 정상 작동하는지 확인**한다. 확인 없이 "완료"라고 보고하지 않는다.
- 정상 작동하지 않거나, 막히거나, 확신이 없으면 **절대 아래 행위를 하지 않는다**:
  - ❌ 거짓으로 작동하는 척 코드를 진행시키기
  - ❌ 문제 부분을 몰래 건너뛰거나 주석 처리로 숨기기
  - ❌ 가짜 데이터·더미 응답으로 성공한 것처럼 위장하기 (Phase stub은 예외이며, 이 경우 반드시 명시)
  - ❌ 에러를 조용히 삼키고(try-catch로 감추고) 넘어가기
  - ❌ "일단 이렇게 두고 나중에 고치자"며 임의로 진행하기
- 대신 **즉시 작업을 중단하고 사용자에게 보고**한다. 보고에는 다음을 포함한다:
  1. 무엇을 하려다 어디서 막혔는지
  2. 에러 메시지나 실패 증상 (있는 그대로, 각색 없이)
  3. 추정되는 원인
  4. 가능한 해결책 후보 (1~3개)
- 그리고 사용자와 함께 **해결책을 검토한 후에** 진행한다. 혼자 판단해서 밀어붙이지 않는다.

**왜 이 규칙이 최우선인가**: 개발이 많이 진행된 뒤에 문제가 발견되면 수정이 매우 어렵거나 불가능하다. 여러 Phase가 쌓인 상태에서 하부의 결함을 고치려면 그 위에 쌓은 모든 것을 다시 손봐야 한다. **작은 문제를 그 즉시 정직하게 드러내는 것이, 나중에 큰 문제로 키우는 것보다 항상 낫다.** 지금 멈추고 보고하는 5분이 나중의 며칠을 아낀다.

### 🚨 규칙 2 — 작게 나눠서 진행 (과부하 방지, 최우선)

> **한 번에 너무 많은 것을 만들지 않는다. 작동하는 작은 조각을 쌓아 올린다.**

- 한 번의 작업(턴)에서 **하나의 태스크만** 수행한다. 여러 파일·여러 기능을 한꺼번에 만들지 않는다.
- 각 Phase는 아래 **Step 단위로 잘게 쪼개서** 진행한다. 한 Step이 끝나고 정상 작동을 확인한 뒤에만 다음 Step으로 간다.
- 새 코드를 작성하기 전에 **"이번 턴에 무엇을, 어느 파일에서, 얼마나 만들지"를 먼저 한 줄로 선언**하고 시작한다.
- 한 번에 200줄 이상의 코드를 쏟아내지 않는다. 크면 나눈다.
- 여러 기능이 필요하면, 목록으로 정리한 뒤 **하나씩 순서대로** 처리하고 각각을 개별 확인·커밋한다.
- **과부하 징후가 보이면 즉시 멈춘다**: 한 응답에서 파일 3개 이상을 동시에 수정하려 하거나, 여러 기능이 서로 얽혀 무엇부터 손대야 할지 불분명하면, 진행하지 말고 사용자에게 "작업을 이렇게 쪼개자"고 제안한다.

**왜 이 규칙이 최우선인가**: 한꺼번에 많은 코드를 만들면 어디서 문제가 생겼는지 찾기 어렵고, 컨텍스트가 넘쳐 앞의 결정을 잊으며, 여러 오류가 뒤섞여 디버깅이 불가능해진다. 작은 단위로 만들고 그때그때 확인하면, 문제가 생겨도 방금 만든 작은 조각만 보면 된다. **개발 속도는 "한 번에 많이"가 아니라 "틀리지 않고 꾸준히"에서 나온다.**

### 🚨 규칙 3 — Phase Gate 통과 없이 다음 Phase 진입 절대 금지 (최우선)

> **현재 Phase의 모든 검증을 100% 통과하기 전에는 다음 Phase를 절대 시작하지 않는다.**

- 각 Phase에는 **자동 검증 · 수동 검증 · 통합 검증** 세 종류의 통과 기준이 있다 (13장 참조).
- 이 세 가지가 **하나도 빠짐없이 전부 통과**해야만 다음 Phase로 넘어간다. **단 하나라도 실패·미확인·불확실하면 진입 금지.**
- Phase를 넘어가기 전, `qa` 에이전트가 아래 **Phase Gate 리포트**를 작성해 사용자에게 제출하고 **명시적 승인**을 받는다:
  ```
  ✅ Phase N Gate 리포트
  - 자동 검증: [통과/실패] — 무엇을 어떤 명령으로 실행해 확인했는지
  - 수동 검증: [통과/실패] — 무엇을 직접 해보고 어떤 결과였는지
  - 통합 검증: [통과/실패] — 전체 흐름 실행 결과
  - 미해결 이슈: [없음 / 목록]
  - 다음 Phase 진입 가능 여부: [가능/불가]
  ```
- **금지 사항** (규칙 1과 연동):
  - ❌ 검증을 건너뛰고 "아마 될 것"이라며 다음 Phase 시작
  - ❌ 실패한 검증을 통과로 표시하거나 조용히 넘기기
  - ❌ "나중에 고치고 일단 진행" (누적되면 수정 불가)
  - ❌ 수동 검증(실측 비교 등)을 사용자에게 요청하지 않고 자동만으로 통과 처리
- 검증이 실패하면 다음 Phase로 가지 말고 **현재 Phase에 머물러 수정**한다. 수정 후 검증을 다시 전부 돌린다.

**왜 이 규칙이 최우선인가**: FITME는 각 Phase가 앞 Phase의 출력에 의존한다. Phase 2 측정이 부정확한 채로 Phase 4 핏 분석을 만들면, 핏 결과도 통째로 틀리는데 그 원인이 Phase 2인지 4인지 알 수 없게 된다. **부실한 토대 위에 쌓을수록 무너뜨리기만 어려워진다.** 각 층이 단단한지 확인하고 다음 층을 올린다.

### 그 외 규칙

4. **Phase 범위 엄수**: 현재 진행 중인 Phase의 코드만 작성한다. 다음 Phase의 로직은 **절대 미리 구현하지 않고 stub(빈 껍데기)으로만** 둔다. "친절하게" 다음 단계를 앞서 만들지 않는다.
5. **완료 파일 보호**: 완료 표시된 Phase의 파일은 **명시적 지시 없이 수정하지 않는다**. 리팩터링이 필요해 보여도 먼저 사용자에게 물어본다.
6. **타입 우선**: 모든 데이터 구조는 `src/types/index.ts`에 먼저 정의하고, 코드는 그 타입을 따른다. 타입 변경이 필요하면 이 CLAUDE.md의 타입 섹션부터 업데이트한다.
7. **컨텍스트 갱신**: 작업이 길어지면(20개 메시지 이상) 이 CLAUDE.md와 `src/types/index.ts`를 다시 읽고 진행한다.
8. **버전 고정**: 아래 명시된 패키지 버전을 임의로 올리지 않는다. 변경이 필요하면 먼저 협의한다.
9. **체크포인트**: 각 Phase(및 큰 Step) 완료 시 반드시 `git commit`으로 저장한다. 커밋 메시지는 `"Phase N-Step M 완료: <요약>"`.
10. **보안**: API 키는 **절대 프론트엔드에 두지 않는다**. 백엔드 `.env`에만 저장하고 `.gitignore`에 등록한다.

## 🎯 1. 프로젝트 개요

### 최종 목표
사용자가 **기준물(신용카드 또는 ArUco 마커)을 몸에 대고 전신 사진을 촬영**하면, AI가 신체 치수를 자동 분석하고, 웹사이트 의류의 사이즈 정보와 비교해 **핏(맞음 정도)을 예측**하며, 최종적으로 **가상 착용 이미지를 합성**해 보여주는 모바일 앱.

### 전체 흐름
```
카메라 촬영(기준물 포함) → 신체 치수 분석 → 의류 사이즈 비교 → 핏 예측 → 이미지 합성
```

### 개발 원칙
- **모바일 우선**: 카메라 필수이므로 모바일 브라우저(PWA) 기반
- **순차 개발**: 5개 Phase를 하나씩, 각 Phase 안에서도 Step 단위로 완성 (규칙 2)
- **AI + 수학 역할 분담**: 척도·왜곡 보정은 OpenCV(수학)가, 신체 부위 인식은 Claude Vision(AI)이 담당

## 📏 2. 측정 방식 — 2단계 모드 (핵심 설계)

FITME는 정확도와 편의성의 트레이드오프를 사용자가 선택하도록 **두 가지 모드**를 제공한다.

### 🟢 간편 모드 (Simple Mode) — 신용카드 기준
- **기준물**: 신용카드·교통카드 (ISO/IEC 7810 국제표준, **85.6 × 53.98 mm**, 전 세계 동일)
- **장점**: 누구나 지갑에 소지, 프린터 불필요, 플라스틱이라 휘지 않음(평면 유지)
- **정확도**: 중상 (호모그래피 보정 적용 시 충분히 실용적)
- **대상**: 대부분의 일반 사용자 (기본 경로)
- **촬영**: 카드를 측정하려는 부위(예: 가슴/배) 정면에 대고 전신 촬영

### 🔵 정밀 모드 (Precise Mode) — ArUco 마커 기준
- **기준물**: 앱이 제공하는 **ArUco 마커** (한 변 크기 고정, 예: 5cm, PDF로 출력)
- **장점**: 서브픽셀 정밀도, ID 식별로 오인식 없음, 다중 마커 배치 가능
- **정확도**: 최고
- **대상**: 프린터 보유 + 최대 정확도를 원하는 사용자
- **촬영**: 출력한 마커를 부위 정면에 대고 촬영. (선택) 정면·측면에 서로 다른 ID 마커 배치 시 둘레 정밀도 상승

### 공통 원리: 호모그래피 변환
두 모드 모두 **"크기를 아는 직사각형이 사진에서 찌그러진 정도로 카메라 각도·거리를 역산"**하는 호모그래피 변환을 사용한다. 차이는 정밀도뿐이며, 코드 구조는 기준물 종류만 바꾸면 되도록 공통화한다.

> **주의**: ArUco(정밀 모드)를 써도 아래 4대 왜곡 중 **깊이·자세·의류 문제는 해결되지 않는다.** 반드시 3장의 왜곡 극복 전략을 함께 적용한다.

## 🔬 3. 왜곡 극복 전략 (정확도 핵심 — 반드시 적용)

A4/카드/마커 어떤 기준물을 써도 2D 사진 측정에는 4대 왜곡이 있다. ArUco는 이 중 일부만 해결하므로, 나머지는 아래 전략으로 보완한다.

### 4대 왜곡과 ArUco의 한계
| 왜곡 요인 | ArUco가 해결? | 보완 전략 |
|-----------|:-------------:|-----------|
| ① 원근 왜곡 (가까운 부위가 크게) | ✅ 해결 | 호모그래피 (ArUco/카드 공통) |
| ③ 렌즈 왜곡 (가장자리 휘어짐) | ✅ 대부분 해결 | 카메라 보정 계수 |
| ② 깊이 차이 (평면 vs 입체 둘레) | ❌ 못 함 | **전략 1·2 필요** |
| ④ 자세·의류 (실루엣 왜곡) | ❌ 못 함 | **전략 1·3 필요** |

### 전략 1 — 촬영 단계에서 왜곡 줄이기
- **기준물을 측정 부위와 같은 평면에 배치**하도록 유도 (원근·깊이 오차 최소화)
- **촬영 거리·각도 가이드**: 화면에 전신 프레임 + "약 2m 거리" 안내, 기기 자이로로 수직 여부 실시간 표시
- **밀착 의류 + 자세 가이드**: 헐렁한 옷·기울어진 자세를 사전 안내로 방지 (④ 대응).
  카메라 화면에 "정확한 실측을 위해 최대한 밀착되는 옷을 입고 측정하세요" 안내 문구 필수 (2-6 실측에서 헐렁한 하의가 다리안쪽 −27cm 오차 유발 확인)
- **(정밀) 정면+측면 2장 촬영**: 서로 다른 ID의 ArUco 마커를 각 평면에 두어 깊이 정보 확보 (② 대응)

#### 촬영 가이드 4층위 (2026-07-16 결정 — 제품 핵심 기능, 구현은 2-8)

배경: 가이드 없이 정확한 촬영 자체가 어렵다는 것이 실제 재촬영 시도에서 확인됐다.
호모그래피는 "마커 평면"만 보정하므로, 몸 전체의 원근 왜곡(머리 확대, 한쪽 몸 부각,
광각 가장자리 늘어남)은 촬영 단계의 **예방·감지**로만 대응할 수 있다.

- **층위 1 — 정적 안내 (촬영 전)**: 밀착 옷 / 약 2m 거리 / 마커를 가슴에 평평하게 /
  전신을 프레임에 꽉 차게 / 카메라를 배꼽~가슴 높이에 수직으로 / 몸을 화면 정중앙에
- **층위 2 — 실시간 오버레이**: 정중앙 실루엣 가이드(광각 가장자리 회피) +
  자이로 표시(좌우 수평 + 앞뒤 기울기 — Phase 1 구현의 확장)
- **층위 3 — 실시간 검증 + 자동 촬영**: 마커 보임 / 크기(measure.py의
  MIN_MARKER_WIDTH_PX) / 중앙 위치 / 가로세로 척도 비율(TILT_RATIO_RANGE, 정면≈1 —
  원본 fixture 실측 0.4201/0.4201이 근거)로 비스듬함 감지 → 조건 충족 시 카운트다운
  자동 촬영. 판정은 백엔드 `POST /check-photo`(2-7c, AI 호출 0)가 담당.
  클라이언트 방식(OpenCV.js vs 서버 폴링)은 2-8에서 결정
- **층위 4 — 촬영 후 피드백**: 미검출·기울어짐 등 구체 사유의 재촬영 안내 +
  랜드마크 좌우 대칭성 검사(2-7 해부학 검증의 확장) — 비대칭이면 신뢰도 하향 + 재촬영 권고

### 전략 2 — 분석 단계에서 수학적 보정
- **호모그래피 변환**: 기준물 네 꼭짓점으로 원근 제거 (OpenCV `cv2.getPerspectiveTransform`)
- **키 캘리브레이션 (2-6 결정, 2-7b 구현 — A안 확정)**: 사용자가 입력한 키
  (UserProfile.heightCm)로 전체 척도를 확정(머리~발뒤꿈치 픽셀 중앙값 대비,
  사진당 척도 1개). **몸 측정은 전 항목 키 척도(A안 — v1/v2/v3 폭 일관성 검증으로
  B안(폭=마커) 대비 우세 확인)**. 마커 척도는 교차 검증(r 판정: ≤5% ok /
  ≤20% 깊이 편향 경고 / >20% 키 입력·마커 배율 의심 → 전 항목 신뢰도 강등)과
  품질 게이트 전용. 호모그래피는 마커 근방(기울기 판정) 전용 — 원거리 외삽은
  v3 폭주(+70.8cm) 실증으로 폐기
- **둘레 추정 개선 (② 대응)**: 정면 너비 → 체형 유형별 타원 계수로 둘레 근사. 측면 사진이 있으면 근사 대신 실측 타원 둘레 계산
- **몸무게 기반 둘레 보정 (2-6 결정)**: 사용자 입력 몸무게(UserProfile.weightKg, 선택)로
  BMI를 구해 타원 깊이 계수를 체형에 맞게 조정
- **정의 보정 계수 (2-6 결정, 2-7b 역산 완료)**: 줄자(곡면을 따라 측정)와 카메라(직선
  투영)의 측정 정의 차이를 부위별 상수로 보정. person01 **v2(밀착 의류) 기준 역산**:
  어깨 곡면 계수 1.1823, 둘레 깊이 계수 chest 0.7681 / waist 0.7273 / hip 0.8331
  (BMI 가감 제외 base. waist는 옵션 A 허리 정의 확정 후 신정의 캐시로 재역산 —
  2026-07-16). ⚠️ 표본 1명 — 일반화 미검증 (문헌 인용 배제 — 13-2).
  v1(헐렁한 옷) 데이터는 여전히 보정 기준으로 쓰지 않음(엣지 케이스용)
- **렌즈 왜곡 교정 (③ 대응)**: 기기 정보 기반 왜곡 계수 적용, 또는 마커 변의 직선성으로 역산

### 전략 3 — 다중 측정·통계로 신뢰도 확보
- **다중 프레임 촬영**: 2~3초간 여러 프레임 캡처 → 측정값 **중앙값** 사용 (흔들림·이상치 제거, Claude Vision 비결정성 완화)
- **해부학적 비율 교차 검증**: 팔길이-키, 어깨-골반 등 정상 비율 범위를 벗어나면 오류 표시 후 재촬영 유도
- **신뢰도 점수 제공**: 각 부위에 high/medium/low 부여. 기준물이 기울거나 의류가 헐렁하면 해당 부위 신뢰도 자동 하향 + 사용자 안내

## 🤖 4. AI 에이전트 설정 (Sub-Agents)

개발 품질을 높이기 위해 Claude Code는 **역할별 서브 에이전트**를 정의하고, 작업 성격에 따라 해당 역할로 전환해 수행한다.

> 사용법: `.claude/agents/`에 정의를 두거나, 작업 시 "지금은 [에이전트명] 역할로 수행해줘"라고 지시한다. **단, 규칙 2에 따라 한 번에 한 에이전트의 한 태스크만 수행한다.**

### 4-1. `architect` — 설계 에이전트
- **책임**: 타입 정의, 파일 구조 설계, Phase 간 인터페이스 계약 수립
- **산출물**: `src/types/index.ts`, 디렉토리 구조, stub 함수 서명

### 4-2. `frontend` — 프론트엔드 에이전트
- **책임**: React 컴포넌트, 카메라 UI, 모드 선택 UI, 촬영 가이드 오버레이, PWA
- **주의**: iOS/Android 카메라 호환성, EXIF 회전, 자이로 수직 표시, 반응형

### 4-3. `backend` — 백엔드 에이전트
- **책임**: FastAPI 서버, OpenCV(ArUco·호모그래피), Claude API 연동, 이미지 처리
- **주의**: API 키 보안, JSON 파싱 방어, 타임아웃·리사이즈, 척도 계산 정확성

### 4-4. `vision` — 측정 알고리즘 에이전트
- **책임**: 기준물 검출(카드/ArUco), 호모그래피, 신체 부위 좌표 추출, 왜곡 극복 3전략 구현
- **원칙**: OpenCV(척도·왜곡)와 Claude Vision(부위 인식)의 역할 분담을 명확히 유지

### 4-5. `qa` — 검증 에이전트
- **책임**: 각 Phase/Step 완료 기준(Definition of Done) 점검, 통합 테스트, 엣지 케이스
- **원칙**: 이 에이전트의 승인 없이 다음 단계로 진입 금지
- **정직성 의무 (규칙 1 연동)**: 완료 기준을 **실제로 실행해서** 통과 여부를 확인한다. 코드가 존재한다는 이유만으로 "통과"로 표시하지 않는다. 하나라도 실패하면 위장하지 말고 **즉시 중단·보고**한다.

### 4-6. `reviewer` — 코드 리뷰 에이전트
- **책임**: 타입 일관성, 보안 취약점, Phase 범위·과부하 규칙 준수 검토
- **원칙**: 커밋 전에 그룹 A·B·C 규칙 위반과 규칙 1·2 위반을 점검한다

### 에이전트 협업 흐름
```
architect (계약 수립)
    ↓
frontend / backend / vision (한 번에 하나씩 — 규칙 2)
    ↓
reviewer (규칙·타입·보안·과부하 점검)
    ↓
qa (Phase/Step Gate 통합 테스트)
    ↓
git commit → 다음 Step
```

## 🔧 5. 하네스 엔지니어링 (Harness Structure)

Claude Code 자신이 오케스트레이터가 되어 에이전트를 조율한다.

### 5-1. 오케스트레이터 책임
- 현재 Phase·Step과 작업 상태 추적
- 적절한 서브 에이전트에게 **하나의 태스크만** 위임 (규칙 2)
- Phase/Step Gate 통과 여부 확인 후 진행
- 에러 발생 시 직전 커밋으로 롤백 결정

### 5-2. 작업 단위 (Task) 구조
각 작업은 아래 형식으로 정의하고 수행한다. **한 번에 하나의 Task만.**
```
Task:
  phase: 현재 Phase 번호
  step:  현재 Step 번호
  agent: 담당 에이전트
  scope: 이번 작업에서 건드릴 파일 (최대 1~2개, 그 외 수정 금지)
  done_when: 완료 판단 기준
  depends_on: 선행 완료되어야 할 Step
```

### 5-3. 과부하 방지 오케스트레이션 (규칙 2 실행 방법)
- 새 Phase 시작 시, 오케스트레이터는 먼저 그 Phase를 **3~6개의 Step으로 분해한 목록**을 제시하고 사용자 확인을 받는다.
- 각 Step은 "한 턴에 끝낼 수 있는 크기"여야 한다. 크면 더 쪼갠다.
- Step 하나를 끝내면 **반드시 정상 작동을 확인**하고, 커밋한 뒤, 다음 Step으로 넘어간다.
- 절대 여러 Step을 한 턴에 몰아서 진행하지 않는다.

### 5-4. 상태 관리
- 진행 상황을 `PROGRESS.md`에 기록 (현재 Phase·Step, 완료 항목, 다음 할 일, 마지막 커밋 해시)
- 각 Step 완료 시 `PROGRESS.md` 업데이트 + git commit

## 📐 6. 공유 타입 정의 (Contract-First) — 코드보다 먼저

**모든 타입은 `src/types/index.ts` 단일 파일에 정의한다.** Phase 1에서 미리 전부 작성한다.

```typescript
// ===== 측정 모드 & 기준물 =====
export type MeasurementMode = 'simple' | 'precise';
export type ReferenceObject = 'card' | 'aruco';

export interface ReferenceInfo {
  type: ReferenceObject;
  realWidthMm: number;   // card: 85.6, aruco: 출력 크기(예: 50)
  realHeightMm: number;  // card: 53.98, aruco: 정사각형이면 width와 동일
  detected: boolean;     // 사진에서 검출 성공 여부
  cornersPx?: [number, number][]; // 검출된 네 꼭짓점 픽셀 좌표
}

// ===== 촬영 품질 판정 (촬영 가이드 층위 3 — 2-7c 산출물) =====
export interface PhotoCheckResult {
  ready: boolean;           // 모든 조건 충족 → 자동 촬영 가능
  reference: ReferenceInfo; // 검출 결과 (미검출이면 detected:false)
  markerSizeOk: boolean;    // 마커 크기 충분 (원거리·저해상도 방지)
  markerCentered: boolean;  // 마커가 화면 중앙 영역에 위치
  tiltOk: boolean;          // 가로세로 척도 비율이 정면 범위(≈1)
  reasons: string[];        // 불충족 사유 (한국어, 사용자 안내용)
}

// ===== 캡처 이미지 (Phase 1 → 2, 1 → 5 공통 규격) =====
export interface CapturedImage {
  base64: string;        // 이미지 데이터 (data: 프리픽스 제외)
  width: number;         // 리사이즈 후 너비 (항상 1080 기준)
  height: number;
  mimeType: string;      // "image/jpeg"
  rotation: number;      // EXIF 보정 후 항상 0
  frames?: string[];     // 다중 프레임(전략 3)용 base64 배열
}

// ===== 사용자 프로필 (정확도 보강 입력 — 2-6 결정) =====
export interface UserProfile {
  heightCm: number;   // 키(cm) — 스케일 캘리브레이션 기준값 (필수)
  weightKg?: number;  // 몸무게(kg) — 둘레 깊이 계수(BMI) 보정용 (선택)
}

// ===== 신체 치수 (Phase 2 산출물) =====
export interface BodyMeasurements {
  height: number;              // 키 (cm)
  shoulder_width: number;      // 어깨 너비
  chest_circumference: number; // 가슴 둘레
  waist_circumference: number; // 허리 둘레
  hip_circumference: number;   // 엉덩이 둘레
  arm_length: number;          // 팔 길이
  inseam: number;              // 다리 안쪽 길이
  torso_length: number;        // 상체 길이
  confidence: Record<string, 'high' | 'medium' | 'low'>;
  mode: MeasurementMode;       // 어떤 모드로 측정했는지
  reference: ReferenceInfo;    // 사용한 기준물 정보
}

// ===== /analyze 응답 (Phase 2-8a — 미검출·실패를 가짜 숫자 없이 전달) =====
export interface AnalyzeStats {
  runs: number;                            // 랜드마크 추출에 성공한 프레임 수
  spreadCm: Record<string, number>;        // 항목별 반복 편차 (cm)
  scale: Record<string, number | string>;  // 사용 척도 요약 (역추적용)
}

export interface AnalyzeResponse {
  ok: boolean;                     // 측정 성공 여부
  reference: ReferenceInfo;        // 기준물 검출 결과 (미검출이면 detected:false)
  measurements?: BodyMeasurements; // ok=true일 때만 존재
  warnings: string[];              // 범위·해부학·척도·대칭 경고 (한국어)
  stats?: AnalyzeStats;            // ok=true일 때만 존재
  error?: string;                  // ok=false 사유 (한국어, 재촬영 안내용)
}

// ===== 의류 스펙 (Phase 3 산출물) =====
// 부위 필드는 전부 선택 — 의류 종류마다 제공 부위가 다르다 (무신사 실증:
// 상의 = 가슴·어깨·소매·총장, 하의 = 허리·엉덩이·허벅지·밑위·총장·밑단).
// 없는 부위를 0 등 가짜 숫자로 채우지 않는다 (규칙 1). 단면(flat) 표기는
// ×2로 둘레 환산해 저장한다 (3-3 결정, 2026-07-17).
export interface ClothingSize {
  label: string;         // 원본 표기: "M", "95", "L", "Free"
  chest_cm?: number;     // 가슴둘레 (단면×2 환산)
  waist_cm?: number;     // 허리둘레 (단면×2)
  hip_cm?: number;       // 엉덩이둘레 (단면×2)
  shoulder_cm?: number;  // 어깨너비 (직선)
  sleeve_cm?: number;    // 소매길이
  length_cm?: number;    // 총장
  thigh_cm?: number;     // 허벅지둘레 (단면×2)
  rise_cm?: number;      // 밑위
  hem_cm?: number;       // 밑단둘레 (단면×2)
  estimated?: boolean;   // true = 실측이 아닌 호칭 기반 근사 (label 변환표 산출)
}

export interface ClothingSpec {
  brand: string;
  url: string;
  category: 'top' | 'bottom' | 'dress' | 'outer';
  productName?: string;        // 표시용 상품명
  fabric?: string;
  stretch?: 'none' | 'low' | 'high';
  sizes: ClothingSize[];       // 정규화된 사이즈 목록
  needsUserInput?: boolean;    // 정규화 불가 표기 존재 → 사용자 실측 입력 요청
  warnings?: string[];         // 정규화 과정 경고 (한국어)
}

// ===== /clothing 응답 (Phase 3-4 — 스크래핑 실패를 크래시 없이 전달) =====
export interface ClothingResponse {
  ok: boolean;
  spec?: ClothingSpec;   // ok=true일 때만 존재
  cached?: boolean;      // 서버 SQLite 캐시 적중 여부 (재조회는 무신사 접속 없음)
  error?: string;        // ok=false 사유 (한국어, 사용자 안내용)
  code?: 'unsupported' | 'not-found' | 'no-size' | 'network'; // 분기용
}

// ===== 핏 스코어 (Phase 4 산출물) =====
export interface FitScore {
  part: string;                       // chest/waist/hip 등
  status: 'tight' | 'good' | 'loose';
  diff_cm: number;                    // 여유(+)/부족(-) cm
  confidence?: 'high' | 'medium' | 'low'; // 해당 부위 측정 신뢰도 전파 (4-1) —
                                          // low면 판정 자체가 불확실 (둘레 3종 등 12장 한계)
}

// ===== 최종 결과 (Phase 4~5 통합) =====
export interface FitResult {
  measurements: BodyMeasurements;
  clothing: ClothingSpec;
  recommendedSize: string;
  scores: FitScore[];
  recommendation: string;      // 자연어 추천 (한국어)
  imageUrl?: string;           // Phase 5 합성 이미지
}
```

### 사이즈 표준화 규칙 (Phase 3→4 최대 위험 지점)
- 브랜드별 제각각 표기("95", "L", "Free", "38")를 **반드시 실측 cm로 정규화**해 `ClothingSize`에 저장
- 변환 테이블은 `server/data/size_conversion.json`으로 관리
- 정규화 불가 표기는 사용자에게 실측값 입력 요청

## ⚠️ 12. 핵심 제약사항

- **2D 사진 기반**: 둘레는 정면 너비 + 타원 근사로 추산한 참고값 (측면 촬영 시 정밀도 상승)
- **절대 정확도 미검증 (표본 1명, 2026-07-16)**: 측정값은 일관성 기준(반복 편차 2cm)으로만
  검증됨. 타원·정의 보정 계수는 person01 v2 역산값(표본 1명, 일반화 미검증)이라
  다른 체형에서 절대 오차가 수 cm 존재할 수 있음. 깊이 편향 사진(근거리, r>1.05)에서는
  폭 기반 둘레가 +5cm 수준으로 과대될 수 있음(v1/v3 실증 — depth_bias 경고로 안내).
  편향 일정성 가정 위에서 Phase 4 핏 예측이 성립하며, 그 가정은 Phase 4 수동 검증에서 확인.
  표본(3명+) 확보 시 절대 정확도(±3cm/±5cm) 검증 예정 (13-2)
- **의류 의존 편향은 보정 불가**: 헐렁한 하의의 가랑이(crotch) 오차처럼 옷에 따라 변하는
  편향은 "일정한 편향"이 아니므로 계수로 못 잡는다 — 밀착 의류 안내(전략 1)로만 완화
- **둘레 3종(가슴·허리·엉덩이)의 반복 편차 ±4cm 수준 (옵션 B, 2026-07-16)**:
  원인은 비전 모델의 실루엣 가장자리 x좌표 추정 노이즈 + 추출 배치 간 드리프트.
  허리 정의 변경(옵션 A)·프레임 수 증가·옷/마커/거리 변경 어느 것으로도 해결되지
  않음이 실증됨. 대응: 다중 프레임 중앙값 보고 + 신뢰도 low 자동 표시 + Phase 4에서
  신뢰도 반영. **길이 항목(키·팔·다리·상체)과 어깨는 이 한계와 무관하게 안정**
  (관절·정점 앵커는 강건, "옷 위 가장자리"만 취약). 향후 완화 후보: ① 세그멘테이션
  기반 윤곽 추출(좌표 질의 대신 픽셀 단위 — 비결정성의 근원 제거, 신규 스택 필요)
  ② 측면 사진 추가 ③ 표본 확보 시 다인 검증으로 산포의 일반성 확인
- **기준물 필수**: 카드 또는 ArUco 마커가 사진에 선명히 보여야 측정 가능
- **ArUco의 한계**: 원근·렌즈 왜곡은 해결하나 깊이·자세·의류 왜곡은 3전략으로 보완해야 함
- **정면 촬영**: 측면·각도 촬영 시 정확도 급감
- **밝은 환경**: 어두우면 기준물·신체 인식률 저하
- **의류 지원 범위**: 사이즈 테이블이 공개된 쇼핑몰만 파싱 가능
- **의류 파싱은 무신사 전용 (2026-07-18)**: 타 쇼핑몰 미지원 — 입력 시 한국어
  안내(unsupported)로 처리. 무신사 **비공식 공개 API** 의존이라 무신사가 구조를
  바꾸면 파손될 수 있음(파싱을 순수 함수로 분리해 두어 오프라인 테스트가 즉시
  감지 — parse_goods/parse_sizes). 다중 쇼핑몰 지원은 향후 과제이며, Phase 3
  Gate 수동 검증도 이에 맞춰 "무신사 상품 3종(카테고리 달리)"으로 조정됨
- **정밀 모드 프린터**: ArUco 마커는 100% 배율 출력 필요 (축소 출력 시 오차)

## ✅ 13. 검증 방법론 (각 Phase 정상 작동 확인법)

각 Phase가 정상 작동하는지는 **자동 · 수동 · 통합** 세 종류의 검증으로 확인한다. 세 가지가 **전부 통과**해야 다음 Phase로 넘어간다 (규칙 3).

### 13-1. 세 가지 검증의 역할

**자동 검증 — 코드가 스스로 확인**
- 명령어 한 줄로 반복 실행 가능한 검사. `npm run build`, 타입 체크, 샘플 입력→출력 범위 검사 등.
- 장점: 빠르고, 나중에 다른 Phase를 건드려 깨졌을 때 즉시 알려줌 (회귀 방지).
- 한계: "말도 안 되는 값"은 걸러도 "정확한 값"인지는 판단 못 함.

**수동 검증 — 사람이 직접 사용**
- 실제 폰으로 만져보거나, 측정 결과를 실측과 비교하는 검사.
- 카메라·UI·이미지 품질처럼 코드로 판정 불가한 영역, 그리고 **측정 정확도**는 이 방법만이 진짜 검증이다.
- **측정 Phase(2)와 핏 Phase(4)는 수동 검증이 필수.** 자동만으로 통과 처리 금지.

**통합 검증 — 전체 흐름 연결**
- Phase가 끝날 때마다 "처음부터 지금까지"를 한 번에 실행해 단계 전환이 매끄러운지 확인.
- 개별 기능이 다 되어도 이어붙이면 깨질 수 있으므로 반드시 별도로 확인한다.

### 13-2. 측정 검증의 핵심 (Phase 2·4) — 일관성 기준 (2026-07-16 방침 전환)

**Phase 2의 통과 기준은 "일관성"이다**: 동일 사진 반복 측정 편차 **2cm 이내** —
단, 적용 범위는 **길이 항목(키·팔·다리안쪽·상체)과 어깨**다. 둘레 3종은 실루엣
가장자리 추정 노이즈로 이 기준을 구조적으로 만족할 수 없음이 실증되어(옵션 A
검증으로 확인, 2026-07-16 옵션 B 결정) 12장 "알려진 한계"로 이월하고, 신뢰도
low 표시 + Phase 4 반영이 담당한다. 절대 정확도(±3cm/±5cm, 3명 검증)는 표본
제약(피험자 1명, 추가 확보 불가)으로 **Gate에서 제외하고 12장 "알려진 한계"와
향후 과제로 이월**한다.

- **왜 일관성인가**: 편향(bias)이 일정하면 Phase 4 핏 예측은 성립한다
  (측정과 사이즈 비교가 같은 기준을 공유해 상쇄됨). Phase 3~5가 통째로 남은
  현 시점에는 일관성 확보가 절대 정확도보다 우선이다.
- **실측 비교는 "참고 기록"**: person01 실측과의 차이를 기록으로 남기되,
  통과/미통과 판정에는 쓰지 않는다 (Phase 4 편향 검증의 기초 자료).
- **계수 정책**: 타원·정의 보정 계수의 역산 캘리브레이션은 **밀착 의류 기준
  데이터 확보 후** 수행한다. person01 반바지 데이터는 보정 기준이 아니라
  "파이프라인 확인 + 나쁜 조건 엣지 케이스"용이다 (2026-07-16 역할 재정의).
  문헌 인용은 배제한다 — 우리 조건(마커 기반·옷 위 측정·정면 너비→둘레)에
  맞는 연구가 사실상 없고, 어정쩡한 인용은 "검증된 척" 착각을 만들어
  규칙 1 위반 소지가 있다.
- **실질 정확도 검증은 Phase 4가 담당**: 본인이 핏을 아는 옷과의 대조
  (Phase 4 수동 검증)가 "편향 일정성" 가정의 실질 검증이다.
- **향후 과제**: 표본(3명+)이 확보되면 절대 정확도(±3cm/±5cm) 검증을 수행한다.
  **구조는 상시 준비됨 (2-7b 후속)**: `fixtures/calibration_set.json`에 대상 등록 후
  `scripts/calibrate_multi.py` 실행 — 대상별 계수 역산·사람 간 산포·Leave-One-Out
  교차 검증(N≥2 자동)을 출력한다. Phase 진행과 무관하게 언제든 실행 가능.
  대상 추가 절차는 `fixtures/README.md` 참조 (v2 촬영 조건 필수 — 엣지 케이스
  사진 등록 금지). 계수 상수 갱신은 자동이 아니라 사용자 결정 후 수동 +
  pytest·verify_27b 재검증.

### 13-3. Phase Gate 진행 규칙 (규칙 3 요약)

```
현재 Phase 개발 완료
    ↓
자동 검증 실행 → 전부 통과?  ─── No ─→ 현재 Phase 수정
    ↓ Yes
수동 검증 (사용자에게 요청) → 전부 통과?  ─── No ─→ 현재 Phase 수정
    ↓ Yes
통합 검증 (전체 흐름) → 통과?  ─── No ─→ 현재 Phase 수정
    ↓ Yes
qa가 Gate 리포트 작성 → 사용자 승인
    ↓ 승인
git commit → 다음 Phase 진입 허용
```

- **어느 단계든 실패하면 다음 Phase로 가지 않고 현재 Phase에 머문다.**
- 수정 후에는 세 검증을 **처음부터 다시** 돌린다 (부분 통과 인정 안 함).
- `qa`는 실제 실행 근거 없이 통과로 표시하지 않는다 (규칙 1).

### 13-4. 검증 자산 관리

- 자동 검증 스크립트는 `server/tests/`와 프론트 테스트에 두고, Phase마다 누적한다 (지우지 않음).
- 테스트용 샘플 사진·정답값은 `server/tests/fixtures/`에 보관.
- 각 Phase Gate 리포트는 `PROGRESS.md`에 기록해 이력을 남긴다.
